# architecture観点

差分がシステム構造・責務境界・依存方向・契約整合性・状態管理の一貫性を壊していないかを検出する。

## 対象外

以下は指摘しない。

- 命名が実体と乖離している問題。`getUser()` が副作用で削除する等。
- 「ついで」の改名や移動、要求外のUI文言変更。
- 生成物と手書きの混在。
- モック、fixture、snapshotが実契約と乖離している問題。
- Fail Fast欠如。silent fallback、多段 `??`、空catch。
- 命名、型、冗長分岐、保守性など局所的な書き方。
- 自然言語のコメントやUI文言のAIプロース。

## REJECTパターン

### 依存方向

| # | パターン | 例 |
|---|---------|-----|
| A1 | 上位層への逆依存 | DomainがInfrastructureをimport、ModelがViewを知る |
| A2 | 汎用層から特定実装へのimport | 抽象化層に `import { uploadToS3 }` と `if (storage === 's3')` |
| A3 | 循環依存 | A → B → A |
| A4 | 内部実装のパブリックAPI露出 | infra層の関数・内部クラスをindexから再エクスポート |

### 責務境界

| # | パターン | 例 |
|---|---------|-----|
| A5 | 責務の混在 | Controllerに業務ロジック、UseCaseにSQL、Componentにデータ取得と表示 |
| A6 | 抽象度の不一致 | オーケストレーション関数の中に低レベル詳細 (接続確立、パス組立て) |
| A7 | 子コンポーネントが親状態を直接変更 | propの関数呼び出しでなく親のstoreを直接書き換え |

### 契約整合性

| # | パターン | 例 |
|---|---------|-----|
| A8 | 契約定義だけ変更し呼び出し元・生成側・利用側が未更新 | 型を変えたがcallerのシグネチャは古いまま |
| A9 | 新引数・新フィールドが呼び出しチェーンで渡されていない | `options.xxx ?? fallback` が常にフォールバック |
| A10 | 型は追加したが利用側の分岐がない | 追加したdiscriminantを判定していない |
| A11 | 共通ヘルパー・adapterを追加したが既存の同種分岐が旧実装のまま | 統一されていない |

### 解決責務の一元化

| # | パターン | 例 |
|---|---------|-----|
| A12 | 設定・provider・パスなどを複数の層で再解決 | 上位と下位が別々に `options.x ?? config.x` する |
| A13 | ログ表示用と実行用で別解決 | 表示値と実行値が乖離しうる |
| A14 | 入力解釈と実行ロジックの混在 | ループ内で毎回 `options ?? config ?? env` を解決 |

### 状態管理

| # | パターン | 例 |
|---|---------|-----|
| A15 | 派生値を独立した状態として保持 | countとdoubleを別々に持つ |
| A16 | 常に同期すべき複数フィールドを別々に保存 | sourceとnormalizedを両方持ち続ける |
| A17 | 副作用の重複・漏れ | 正常系・失敗系・中断系でreserve/releaseの対称性が崩れる |
| A18 | 中断後の副作用開始 | timeout/cancel後に新しいtool callや外部通知 |

## 検出手順

1. diff内の `import` / `require` を確認し、汎用層 → 特定実装、下位層 → 上位層の依存がないか確認する。
2. 新規追加された関数・型・エクスポートが、レイヤの正しい位置にあるか確認する。
3. 型・スキーマ・APIシグネチャの変更に対し、callerが同じdiffで更新されているか確認する。
4. 新パラメータ・新フィールドを追加した場合、呼び出しチェーン全体をgrepして末端まで渡されているか確認する。
5. 状態を扱う変更 (reducer, store, hook, service) は、正常・失敗・中断経路の対称性を確認する。

## 出力ルール

- 責務の混在を指摘する場合は「どのレイヤに移すべきか」も具体的に示す。
- 契約不整合の場合、更新すべきファイルを列挙する。
