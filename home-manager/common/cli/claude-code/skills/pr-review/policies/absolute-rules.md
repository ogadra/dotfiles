# absolute-rules観点

差分に「勝手に足された互換レイヤ」「サイレントなフォールバック」「明示なしのデフォルト」「Fail Fastの欠如」が入っていないかを検出する。この観点は例外を認めず、REJECTのみで判定する。

## 対象外

以下は指摘しない。

- 自然言語のコメント。「以前は〜」型の経緯、`// deprecated` のラベル、説明コメント。
- 握りつぶし以外のエラー処理品質。catchでの詰め替え、エラーメッセージの不明瞭さ。
- 新パラメータの配線忘れ、依存方向、責務境界などのアーキテクチャ整合性。
- 命名、型、冗長分岐など局所的な書き方。
- テストの不足、モック品質。

## REJECTパターン

### 後方互換

| # | パターン | 例 |
|---|---------|-----|
| AR1 | 明示要求のない後方互換実装 | 旧関数名の残置、旧値の変換マッピング、`@deprecated` エイリアス |
| AR2 | 削除されたはずの旧関数・旧エクスポートの残存 | rename後も旧名を再エクスポート |
| AR3 | 代替経路の勝手な追加 | 主経路の失敗時に別ルートを試す実装 |

### Silent fallback / Fail Fast欠如

| # | パターン | 例 |
|---|---------|-----|
| AR4 | 必須データへのフォールバック | `user?.id ?? 'unknown'`, `os.getenv("FOO", "default")` |
| AR5 | サイレントスキップ | `if (!x) return` で本来エラーの状態を握りつぶす |
| AR6 | try/catchで空値返却 | `try { … } catch { return null }`, `catch { return '' }` |
| AR7 | 多段フォールバック | `a ?? b ?? c ?? d` |
| AR8 | デフォルト引数によるフォールバック | 全呼び出し元が省略している `def f(x="…")` |
| AR9 | 非同期経路の握りつぶし | `.catch(() => {})`, `void promise` |
| AR10 | 空catchでエラー握りつぶし | `try { … } catch (e) {}`, `except: pass` |

## 検出手順

1. diff内の `??`, `||`, `try/catch`, `catch (…) {}`, `= "default"`, `= None`, `getenv(.*,`, `getOrElse`, `orElse`, `.catch(`, `void ` を検索する。
2. それぞれについて次を確認する。
   - 必須データか。必須ならREJECT (AR4)。
   - 全呼び出し元がデフォルトに任せているか。任せているならREJECT (AR8)。
   - 上位から値を渡す経路があるか。なければREJECT。
3. rename・削除された旧関数・旧エクスポートが残存していないかdiff全体で確認 (AR2)。
4. 主経路の失敗時に別ルートを試す分岐を確認 (AR3)。

## 出力ルール

- 該当行と、なぜそのパターンに当たるかを1行で示す。
- 修正案は「エラーをraise」「必須引数化」「呼び出し元から渡す」「旧コードを削除」のいずれかを具体的なファイル・関数名で提示する。
