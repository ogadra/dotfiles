# testing観点

差分にテストが不足していないか、既存テストの品質が保たれているかを検出する。モック・fixture・snapshotが実契約と乖離していないかもこの観点で扱う。

## 対象外

以下は指摘しない。ただしテスト名の文言は対象にする。

- プロダクトコード側の命名、型、冗長、リソース管理などの実装品質。
- 依存方向、責務境界などのアーキテクチャ整合性。
- テストコメントのAIプロース。
- テストコードに混入した機密以外のセキュリティ問題。

## REJECTパターン

### カバレッジ

| # | パターン | 例 |
|---|---------|-----|
| T1 | 新しい振る舞いにテストがない | プロダクトコードのみ追加でテストゼロ |
| T2 | バグ修正にリグレッションテストがない | fixコミットでテスト追加なし |
| T3 | 振る舞い変更にテスト更新なし | 期待値の更新がない |
| T4 | 副作用・状態遷移の失敗経路が未検証 | 正常系のみ |
| T5 | 共通化・adapter追加で新経路のみテスト、既存の同種分岐は未検証 | |
| T6 | パーサー・設定境界で欠落値・不正shapeの検証欠落 | 型どおりの入力のみ |
| T7 | 到達経路の検証欠落 | 画面・機能を追加したがroute / メニュー / リンクからの到達確認なし |
| T8 | UIライブラリ主要コンポーネントの実マウント欠落 | DataGridやChartを導入したが浅いモックのみ |

### テスト品質

| # | パターン | 例 |
|---|---------|-----|
| T9 | 実装の詳細をテスト (振る舞いではなく内部呼び出しを検証) | private関数のスパイ |
| T10 | テスト間でfixtureを共有し変更 | mutableな共有オブジェクトを書き換え |
| T11 | モック・fixture・snapshotが実契約と乖離 | 型は合うがエラー型・欠落値・冪等性・返り値shapeが実装と違う |
| T12 | 空catchと同じ「呼ばれたことだけ確認」 | 引数・副作用を検証しない `expect(fn).toHaveBeenCalled()` |
| T13 | 完全一致文字列の不在で禁止値を判定 | 順序・大小文字・空白違いで漏れる |
| T14 | 固定sleepでE2Eタイミング合わせ | `await sleep(3000)` |
| T15 | 本番APIを叩くE2E | mockなし |
| T16 | 非実行資産のテスト | READMEの見出し・章構成を検証 |
| T17 | 置換された旧仕様の不在確認だけを追加 | 旧バッジが表示されないことだけを検証 |
| T18 | 時刻・乱数依存で再現性なし | `new Date()` を固定せずassert |
| T19 | テスト対象のコア処理をモック | E2Eの意味が消える |

### 契約変更のカバレッジ

| # | パターン | 例 |
|---|---------|-----|
| T20 | 新オプションで未設定・不正・継承・非継承の分岐がない | 正常系のみ |
| T21 | 主実行経路と表示・検証入口で契約が同一である確認がない | |
| T22 | チェーンを通じて末端まで伝搬する新引数の結合テストがない | |

### 警告どまり

| # | パターン | 判定 |
|---|---------|------|
| T23 | エッジケース・境界値テストの不足 | Warning |
| T24 | ユニットで足りるものをE2Eで書く | Warning |
| T25 | factoryを使わず巨大fixtureを直書き | Warning |
| T26 | 実装の詳細に一部依存するが振る舞いテストの範疇に留まる | Warning |

## 検出手順

1. diff内の非テストファイル追加行に対し、対応するテストファイルの追加・更新があるか確認する。
2. `describe`, `it`, `test`, `expect`, `assert` の追加がプロダクトコードの変更をカバーしているか確認する。
3. モック定義 (`jest.mock`, `vi.mock`, `MagicMock`) が対象APIの実型・実契約と合致するか確認する。特にエラー型・欠落値・返り値のshapeの乖離を確認する。
4. `sleep`, `setTimeout`, `Date.now`, `Math.random` がテスト内で決定的でないまま使われていないか確認する。
5. 追加された新機能・新画面に対してroute / メニュー / 入口からの到達検証が入っているか確認する。

## 出力ルール

- 「テストがない」だけでなく、追加すべきテスト観点 (未設定・不正・境界・失敗経路のどれか) を具体的に示す。
- 対応するテストファイルの推奨パスを1つ提示する。
