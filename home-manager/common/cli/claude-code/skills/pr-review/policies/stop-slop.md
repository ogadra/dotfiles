# stop-slop観点

diffに含まれる自然言語 (コメント、docstring、README、コミット説明、CHANGELOG、UI文言、ログ文言、テスト名) からAI生成パターンと不要コメントを検出する。**コードの中身ではなく、自然言語だけを対象にする**。

## 対象外

以下は指摘しない。

- コードの実装ロジック、命名、型、冗長分岐など、自然言語ではないコード部分。
- 削除されたはずの旧関数・旧エクスポートの残置。
- テストの過不足、モック品質。
- セキュリティ、アーキテクチャ整合性。

## REJECTパターン

### 経緯・履歴コメント

| # | パターン | 例 |
|---|---------|-----|
| S1 | 「以前は〜」型のリビジョン履歴 | `// 元はXを返していたがYに変更`, `// 旧APIはxxxを返していた` |
| S2 | 括弧補記の追記 | `// バリデーション（追加）`, `// PR #123で修正` |
| S3 | チケット/PR番号への言及 | `// closes #42`, `// See TICKET-1234` |
| S4 | コメントアウトされた旧実装 | `/* 旧実装 */` |
| S5 | `// deprecated`, `// legacy`, `// backward` などのラベル | |

### 説明コメント (What/How)

| # | パターン | 例 |
|---|---------|-----|
| S6 | Whatを書いた説明コメント | `// user idを取得する`, `// increment counter` |
| S7 | Howを書いた説明コメント | `// 配列をmapで回す` |

Why (非自明な理由・ハック・特殊定数の根拠) は残す。それ以外の説明は削除。

### AI特有のプロース (日本語/英語共通)

| # | パターン | 例 |
|---|---------|-----|
| S8 | 命題型H2 (README/docs) | `## AはBだ`, `## Design is essential` |
| S9 | 二項対比の乱用 | `AではなくBだ`, `Not X, it's Y` |
| S10 | false agency | `The decision emerges`, `データが示している` |
| S11 | 三項並列 | `3つの原則`, `three principles` |
| S12 | 偏愛語・必殺技造語 | 「本質」「解像度」「営み」「文脈」「essentially」「fundamental」を段落に複数 |
| S13 | 全角ダッシュ `──` | markdown・コメント内 |
| S14 | 中黒並列A・B・C | `foo・bar・baz` の羅列 |
| S15 | 装飾絵文字の均等撒布 | 段落末に 🚀 🎯 ✨ 💡 |
| S16 | 過剰なカタカナ | `コンテキスト`, `アライメント`, `コミュニケーション` |
| S17 | 「ケースバイケース」で結論回避 | `場合による`, `it depends` だけで終わる |
| S18 | 過剰な副詞・強調 | `非常に`, `めっちゃ`, `essentially`, `basically`, `simply` |
| S19 | 英数字と日本語の間の半角空白 | 半角英数字と日本語 (かな・漢字) の境界に半角スペース。 |

## 検出手順

1. diff内の追加行のうち、以下のいずれかに該当するものだけ抽出する。
   - コメント行 (`//`, `#`, `--`, `/* */`)
   - docstring / JSDoc
   - `.md`, `.mdx`, `.rst`, `.txt` の追加行
   - 文字列リテラルのうち、UI文言・ログメッセージ・エラー文言・テストの `describe` / `it` 名
2. 各行を上表のパターンに照らす。
3. コードの中身 (変数名・関数名) は対象外。ただしテスト名やUI文字列はプロースなので対象。

## 出力ルール

- 対象箇所を丸ごと引用し、書き換え後の1案を提示する。
- 検出できたパターン番号 (S1〜S19) をcategoryに入れる。
